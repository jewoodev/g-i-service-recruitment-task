# 설계 질문 답변서

## 질문 1

서울시 버스 7,000대가 동시에 접속한다고 가정하자. 현재 구현한 구조에서 어떤 문제가 발생할 수 있는지 설명하고, 실제 서비스로 확장하기 위해 어떤 방식으로 개선할 것인지 설명하시오.

### 답변

현재 구조는 단일 Spring Boot API와 MySQL에 요청이 집중된다. 7,000대 차량이 위치와 이벤트를 주기적으로 전송하고 관제 화면도 polling하면 DB 쓰기 부하, 최신 위치 조회 부하, API 서버 병목이 발생할 수 있다. 실제 서비스에서는 차량 수집 API를 분리하고 Kafka로 위치/이벤트를 비동기 처리한다. 최신 위치는 Redis나 별도 read model에 저장하고, 관제 화면은 WebSocket 또는 SSE로 변경 사항만 전달한다.

## 질문 2

향후 버스 전면 블랙박스, 후면 블랙박스, 내부 CCTV 영상이 추가된다면 어떤 방식으로 저장·전송·관제할 것인지 설명하시오.

### 답변

영상 원본은 DB에 저장하지 않고 미디어 서버나 Object Storage에 저장한다. 차량 단말은 영상을 스트리밍 서버로 전송하고, 관제 화면은 HLS나 WebRTC로 재생한다. DB에는 버스 ID, 카메라 유형, 스트림 URL, 저장 파일 위치, 수신 상태, 녹화 시간 같은 메타데이터만 저장한다. 사고나 민원 이벤트가 발생하면 해당 시각의 영상 구간을 이벤트와 연결해 검색할 수 있게 설계한다.
