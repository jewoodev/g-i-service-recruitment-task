# 백엔드 요구사항

이 문서는 `business-description.md`, `recruitment-task.md`를 기준으로 서울 버스 관제 시스템 MVP에서 백엔드가 책임져야 하는 요구사항을 정리한다.

## 목표

관제 담당자가 웹 화면에서 현재 운행 중인 버스의 상태를 확인할 수 있도록 버스, 노선, 정류장, GPS 위치, 이벤트, 통신 상태 데이터를 제공한다.

이번 MVP는 실시간 영상 관제 플랫폼의 첫 단계다. 실제 영상 저장, 전송, AI 분석 구현보다는 관제 화면을 구성하기 위한 데이터 모델링과 조회 API 구현이 핵심이다.

## 기능 요구사항

### 버스 목록 조회

- 버스번호를 조회할 수 있어야 한다.
- 노선명을 조회할 수 있어야 한다.
- 현재속도를 조회할 수 있어야 한다.
- 현재상태를 조회할 수 있어야 한다.

### 버스 상세 조회

- 버스번호를 조회할 수 있어야 한다.
- 노선명을 조회할 수 있어야 한다.
- 현재속도를 조회할 수 있어야 한다.
- 현재상태를 조회할 수 있어야 한다.
- 마지막 통신시간을 조회할 수 있어야 한다.

### 지도 표시용 위치 조회

- 버스의 최신 GPS 위치를 조회할 수 있어야 한다.
- 지도에 표시할 수 있도록 위도와 경도를 제공해야 한다.
- 전체 운행 버스의 최신 위치를 조회할 수 있어야 한다.
- 특정 버스의 최신 위치를 조회할 수 있어야 한다.

### 최근 이벤트 조회

- 급정거, 급가속, 충격 등 최근 이벤트를 조회할 수 있어야 한다.
- 이벤트 발생 버스, 이벤트 유형, 심각도, 발생 시각을 제공해야 한다.
- 이벤트 발생 위치가 있는 경우 위도와 경도를 제공해야 한다.

### 차량 통신 상태 표시

- 마지막 통신시간을 기준으로 버스의 통신 상태를 계산해야 한다.
- 마지막 통신시간이 현재 기준 5분 이하이면 `ONLINE`으로 판단한다.
- 마지막 통신시간이 현재 기준 5분을 초과하면 `OFFLINE`으로 판단한다.

### Mock Data 제공

- 실제 데이터가 제공되지 않으므로 더미 데이터를 직접 생성해야 한다.
- 더미 데이터는 버스 목록, 버스 상세, 지도 표시, 최근 이벤트 화면을 확인할 수 있을 만큼 포함해야 한다.

## 데이터 요구사항

다음 정보를 저장할 수 있는 데이터 구조를 설계한다.

| 도메인 | 필요 데이터 |
| --- | --- |
| 버스 | 버스번호, 운행 노선, 운행 방향, 마지막 통신시간 |
| 노선 | 노선번호, 노선명, 출발지, 도착지 |
| 노선 방향 | 상행/하행, 방향별 출발지, 방향별 도착지 |
| 정류장 | 정류장명, 위치 좌표 |
| 노선별 정류장 순서 | 특정 노선 방향에서 정류장이 몇 번째인지 |
| GPS 위치 | 위도, 경도, 속도, 수집 시각 |
| 이벤트 | 이벤트 유형, 심각도, 발생 버스, 발생 위치, 발생 시각 |
| 카메라 상태 | 카메라 유형, 수신 여부, 마지막 수신 시각 |

## API 요구사항

프론트엔드가 관제 화면을 구성할 수 있도록 다음 조회 API를 제공한다.

| API | 목적 |
| --- | --- |
| `GET /api/v1/buses` | 버스 목록 조회 |
| `GET /api/v1/buses/{busId}` | 버스 상세 조회 |
| `GET /api/v1/buses/positions/latest` | 지도 표시용 전체 버스 최신 위치 조회 |
| `GET /api/v1/buses/{busId}/positions/latest` | 특정 버스 최신 위치 조회 |
| `GET /api/v1/events/recent` | 최근 이벤트 조회 |

## 비기능 요구사항

- 데이터 모델은 ERD 또는 문서로 설명할 수 있어야 한다.
- README에는 실행 방법과 기술 선택 이유를 포함해야 한다.
- 설계 문서에는 데이터 모델, 시스템 구조, 확장 방안을 포함해야 한다.
- Docker 실행 환경을 제공하면 가산점에 해당한다.
- WebSocket 또는 Polling 기반 실시간 갱신 구조를 제공하면 가산점에 해당한다.
- 7,000대 버스가 동시에 접속하는 상황에서 발생할 수 있는 문제와 개선 방안을 설명해야 한다.
- 향후 영상 데이터가 추가될 때 저장, 전송, 관제 방식을 설명해야 한다.

## MVP 구현 범위

이번 백엔드 구현에서는 다음 범위를 우선한다.

- 버스, 노선, 정류장, GPS 위치, 이벤트 중심의 도메인 모델 설계
- 데이터베이스 스키마와 Mock Data 구성
- 조회 전용 Application Service 구현
- REST API 구현
- 마지막 통신시간 기반 `ONLINE` / `OFFLINE` 계산
- API 통합 테스트
- 실행 방법, 기술 선택 이유, 확장 방안 문서화

## 확장 설계 대상

다음 항목은 과제의 필수 구현보다는 향후 확장 방향으로 다룬다.

- 실시간 영상 스트리밍
- 영상 저장소 연동
- 안전운전 분석
- 사고 탐지
- 승객 안전 분석
- 유실물 검색
- 민원 자동 리포트 생성
- AI 기반 이벤트 분석
- 날씨와 교통 상황 외부 API 연동
